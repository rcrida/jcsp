## Architecture Overview

This is a Constraint Satisfaction Problem (CSP) solver library implementing classic AI algorithms. The core flow is: define a `ConstraintSatisfactionProblem` (variables + domains + constraints), then call `Solver.Factory.INSTANCE.createSolver(csp).getSolutions()` to get a lazy `Stream` of `Assignment` solutions.

### Core Abstractions

- **`Variable`** — immutable identifier; created via `Variable.Factory`
- **`Domain`** — base interface for all domains; defines `contains()`, `isEmpty()`, `size()`, and default `isSingleton()` (via `size() == 1`); `singleValue()` is abstract (discrete domains default it via `stream()`, bounded domains implement it directly)
- **`DiscreteDomain<T> extends Domain<T>`** — enumerable domains; adds `stream()`, `toList()`, and `toBuilder()` (with inner `Builder<T>` interface). Default `singleValue()` uses `stream()`. Code that needs to enumerate values should be typed to `DiscreteDomain`, not `Domain`
- **`SetDomain<T> extends DiscreteDomain<T>`** — intermediate interface for all set-backed discrete domains; declares `values() → Set<T>` and provides default implementations of all `DiscreteDomain` methods via that single accessor. Concrete implementors (`DomainObjectSet`, `IntRangeDomain`, `EnumDomain`, `BooleanDomain`, `AssignedDomain`, `AssignmentDomain`) are all records that implement `SetDomain<T>` directly. Provides static helpers `domainEquals`/`domainHashCode` so cross-type equality works: two `SetDomain` instances with the same `values()` set are equal regardless of concrete record type. `DefaultBuilder` inner class delegates `build()` to `DomainObjectSet`. `IntRangeDomain` and `EnumDomain` store values in a `LinkedHashSet` (preserving ascending / declaration order) so `stream()` is deterministic and the solver's first-solution heuristic is stable.
- **`BoundedDomain<T extends Number & Comparable<T>>`** — `Domain` extension for non-enumerable continuous ranges; `getMin()`/`getMax()`/`withBounds(newMin, newMax)`. `IntervalDomain` is the sole implementation: a record with `double min, double max` components; `size()` returns `1` for a singleton interval and `Integer.MAX_VALUE` otherwise. `SumConstraint`, `LinearConstraint` (with a `Double` bound), `UnaryComparatorConstraint`, `BinaryComparatorConstraint`, `BinaryOffsetConstraint`, `AbsoluteDifferenceConstraint`, `DivisionConstraint` (positive domains only), `LexConstraint`, `CumulativeConstraint`, `MaxConstraint`, `MinConstraint`, `ProductConstraint` (positive domains only), `DiffnConstraint` (origin variables), and `IncreasingConstraint`/`DecreasingConstraint` support `BoundedDomain` variables; `ConstraintSatisfactionProblem`'s build-time validation rejects any other constraint type referencing one with `IllegalArgumentException`. `BinaryComparatorConstraint`, `BinaryOffsetConstraint`, and `AbsoluteDifferenceConstraint` all narrow via `NumericBounds.narrow` rather than `BoundedDomain.withBounds` directly, so a plain discrete/discrete pair gets real bounds propagation (value deletion) too, not only mixed discrete/bounded ones. `BinaryComparatorConstraint` is the one exception with a caveat: it's generic over any `Comparable<T>` (backing `IncreasingConstraint`/`DecreasingConstraint`'s binary decomposition, which can order non-numeric types), so a plain discrete/discrete pair only gets this treatment if its values are actually `Number`s — checked once via a single-element peek, since both sides share the same type parameter — otherwise propagation stays a no-op and the ordering is left entirely to `isSatisfiedBy` plus AC3. `IncreasingConstraint`/`DecreasingConstraint` themselves (2026-07-18) don't have this restriction: their own direct `Propagatable` implementation (see below) narrows via `Comparable<T>` natively — `BoundedDomain#getMin`/`getMax`/`withBounds` are already `T`-typed — rather than converting through `NumericBounds`/`double`, so a non-numeric chain (`String`, enum) gets real propagation too, not just a numeric one
- **`Assignment`** — immutable mapping of variables to values; validated against domains and constraints
- **`Constraint`** / `UnaryConstraint` / `BinaryConstraint` / `NaryConstraint` — hierarchical constraint interfaces; each checks `isSatisfiedBy(Assignment)`
- **`ConstraintSatisfactionProblem`** — aggregates variables, domains, constraints; analyzes graph structure (tree/cyclic, connected components, cutsets); `isFullyDetermined()` returns true when every variable's domain is a singleton. Carries a `nogoods: Set<NogoodConstraint>` field (2026-07-12) separately from its structural `constraints`/`ConstraintGraph`; `withNogoods(Set<NogoodConstraint>)` replaces it and reuses the existing `ConstraintGraph` reference untouched, since a `NogoodConstraint` never contributes to neighbours, binary decomposition, or cycle/connectivity analysis. `getConstraints()` returns a precomputed flat union of the structural constraints and `nogoods`, cheap relative to the graph analysis it deliberately bypasses. `nogoods` and this derived union are excluded from `equals`/`hashCode` (learned nogoods are accumulated search knowledge, not part of the problem's identity); `constraintGraph` itself, by contrast, *is* included, so two CSPs with the same variables but different structural constraints are correctly distinct.
  - **Nogood-merge caching (2026-07-16)** — computing that flat union used to happen unconditionally in the constructor on every call, confirmed via `NogoodPropagationBenchmark` to be the dominant per-node cost in long searches (it scales with total nogood count, up to `20 * variableCount`). A `nogoodMergeCache` field (an `AtomicReference<NogoodMergeCache>`, `NogoodMergeCache` a private record of `(nogoods reference, merged Set)`) is threaded through `toBuilder()` exactly like `constraintGraph` already is; the constructor reuses the cached merge when the incoming `nogoods` is the *exact same* `Set` reference as last time (a cheap identity check — always correct on a miss, since it just falls back to a fresh merge). `withNogoods(newNogoods)` also short-circuits to `return this` when `newNogoods == this.nogoods` (nothing learned since this CSP was built — the common case across ordinary search descent), skipping the builder/constructor/merge entirely. The `nogoods` builder parameter dropped `@Singular` for this (it was silently double-copying internally on top of whatever the caller already copied); a hand-written `.nogood(NogoodConstraint)` builder method replaces it for the handful of low-volume construction call sites. None of this touches `ConstraintGraph`, which stays deliberately nogood-unaware per its own "immutable structural representation" contract — see `NogoodStore` below for the caller-side half of this fix.
- **`SolverLimits`** — immutable configuration for node and time limits (`ofNodes`, `ofTime`, `of`, `unlimited`); also holds mutable runtime state (`AtomicReference<Statistics> limitHitStats`, excluded from `equals`/`hashCode`/`toString` via Lombok annotations). `markLimitReached(Statistics)` is called by search methods on first limit detection (CAS); `isLimitReached()` / `getLimitHitStatistics()` / `resetLimitReached()` are called by `BoundSolver.getSolution()`. `deadlineNanos()` returns `Long.MAX_VALUE` for unlimited; `isTimeLimitExceeded` uses subtraction (`nanoTime - deadline >= 0`) for overflow-safe comparison. Set via `SolverConfig.builder().limits(...)` (see below); the no-arg `createSolver` overloads delegate to `SolverConfig.builder().build()`, i.e. `SolverLimits.unlimited()`.
- **`SolverConfig`** (2026-07-16, `solver` package) — `@Value @Builder` bundling `createSolver`'s configuration knobs (`limits`, `nogoodLearningEnabled`, `statistics`) behind one type with `@Builder.Default`s, replacing what used to be a growing set of `createSolver` overloads (one raw parameter per knob) that didn't scale as more got added. `createSolver(csp, SolverConfig)` / `createSolver(csp, objective, SolverConfig)` are the two true abstract methods on `Solver.Factory`; the no-arg and objective-only overloads default to `SolverConfig.builder().build()`. This was a breaking API change (replaced the prior `createSolver(csp, SolverLimits)`/`createSolver(csp, objective, SolverLimits)` signatures outright rather than deprecating them, a deliberate choice given the accumulating-overloads problem it fixes). `nogoodLearningEnabled` (originally a `conflictExplainer` field, simplified to a plain `boolean` on 2026-07-18 when `ConflictExplainer` was merged into `Inference` — see the "Conflict explanation" section below) now affects **both** chains: `Solver.Factory`'s shared `nogoodLearningInference(SolverConfig)` helper picks between `FULL_PROPAGATION_INFERENCE` (`true`, disabling CDCL entirely — no explanation computation, no accumulation — for problem shapes where learned nogoods rarely get reused) and `Inference.withoutReasonTracking(FULL_PROPAGATION_INFERENCE)` (`false`), and both `DomWdegLubySearch` and `BranchAndBoundSolver` are handed whichever `Inference` that helper returns — `BranchAndBoundSolver` gained its own `NogoodStore` the same day (2026-07-18), so it is no longer true that only the satisfaction chain learns nogoods.
- **`LimitExceededException`** — unchecked exception thrown by `BoundSolver.getSolution()` (satisfaction chain only) when a node or time limit is exceeded; carries a `Statistics` snapshot captured at the point of detection. Distinguishes limit-hit from genuine UNSAT (`Optional.empty()`). `getSolutions()` truncates the stream silently — callers who want partial results should consume the stream directly
- **`BoundSolver`** — public API returned by `Solver.Factory.createSolver(csp)` and `createSolver(csp, objective)`; wraps a built chain with the CSP already bound. `getSolutions()` and `getSolution()` are both abstract (no default). The satisfaction factory returns an anonymous class that resets `limits.resetLimitReached()` before each search, calls `getSolutions().findFirst()`, then throws `LimitExceededException` if `limits.isLimitReached()`. The optimization factory returns an anonymous class that overrides `getSolution()` to use `reduce((a,b) -> b)` — limits truncate the stream silently and the best partial result is returned
- **`PropagationResult`** — record (`consistency` package) returned by `Propagatable.propagateWithReasons`: `@Nullable updatedDomains` paired with a `@Nullable NogoodConstraint reason` (a field/parameter, so `@Nullable` per this project's convention — `Optional` is reserved for return types; see `explainInfeasible` below). `feasible(domains, reason)` / `infeasible(reason)` are the two static factories; `isInfeasible()` is `updatedDomains == null`. A `null` reason means the propagator hasn't implemented explanation and the caller falls back to the full assignment as the nogood. Powers `FixpointConsistency`/`AC3`/`NogoodFixpointConsistency`'s own `applyWithReason` overrides (see `ConstraintConsistency` below) and, transitively, `Inference#applyWithReason` (see `DomWdegLubySearch`/`BranchAndBoundSolver` below)
- **`NogoodStore`** — `@Value` class (`assignments` package) that accumulates learned nogoods during backtracking search, modelled as actual `NogoodConstraint`s (`constraints.nary` package, an interface — see below) rather than a separate variable-value map matched against the search's explicit assignment; `record(NogoodConstraint)` adds a nogood directly to an internal `Set` (`ConcurrentHashMap.newKeySet()`), `apply(csp)` returns `csp.withNogoods(...)` — a no-op when none have been recorded. `ConstraintSatisfactionProblem` carries its own `nogoods` field independently of its structural constraints/`ConstraintGraph` (see `withNogoods` in Core Abstractions above), so `apply` never triggers constraint-graph recomputation — a `NogoodConstraint` never contributes to neighbours, binary decomposition, or cycle/connectivity analysis. The set is deduplicated and capped at `maxNogoods` (`forProblem` scales this to `20 * variableCount`, floored at 50, both instance constants): once `record` pushes the size over the cap, the largest-arity nogoods are evicted first — a full-assignment fallback nogood (see `Inference#applyWithReason`'s default, below) is bounded by search depth rather than a constant, and is the least likely of the two to ever be reused. See `DomWdegLubySearch`/`BranchAndBoundSolver` below for how it's wired into search — both terminal solvers now share this same wiring.
  - **Snapshot caching (2026-07-16)** — `apply` used to call `Set.copyOf(nogoods)` unconditionally on every invocation (i.e. every node); it now caches that snapshot in an `AtomicReference<Set<NogoodConstraint>>`, invalidated (set to `null`) only inside `record`/eviction when the set actually mutates — once per learned nogood, not once per node. This is what lets `ConstraintSatisfactionProblem`'s own merge cache (see above) get a stable `Set` reference to key its identity check on across the many nodes visited between nogood-learning events. (An earlier `AtomicReference<ConstraintSatisfactionProblem>` cache here, unrelated to this one, existed solely to dodge constraint-graph rebuilding and was removed entirely once nogoods moved onto the CSP directly (2026-07-12) — that rebuild concern and this one are different costs.)

### Solver Chain (Decorator Pattern)

`Solver.Factory.INSTANCE` builds two distinct chains, each returning a `BoundSolver` with the CSP already bound:

**Satisfaction** (`createSolver(csp)`): `NodeConsistency → PropagationFixpoint(snap=true) → IndependentSubproblems → TreeDecomposition → CutsetConditioning → TreeSolver / DomWdegLubySearch`

**Optimization** (`createSolver(csp, objective)`): `NodeConsistency → PropagationFixpoint(snap=false) → BisectionConditioning (continuous only) → BranchAndBound`

Key decorators:

1. **`NodeConsistentSolver`** — prunes domains via node consistency
2. **`PropagationFixpointSolver`** — runs UnaryComparatorConstraint/BinaryComparatorConstraint/BinaryOffsetConstraint/AbsoluteDifferenceConstraint interval bounds clipping, AC3, NogoodConstraint unit propagation, AllDiff GAC (Régin 1994), SumConstraint bounds propagation, LinearConstraint bounds propagation, CountConstraint value propagation, InverseConstraint arc consistency, AmongConstraint value-set propagation, AtLeastNConstraint/AtMostNConstraint/AtMostOneConstraint/ExactlyOneConstraint boolean forcing, CumulativeConstraint timetabling propagation, GlobalCardinalityConstraint value propagation, LexConstraint bounds propagation, MaxConstraint bounds propagation, MinConstraint bounds propagation, ProductConstraint bounds propagation (positive domains), DivisionConstraint bounds propagation (positive domains), NaryTuplesConstraint table GAC, CircuitConstraint Hamiltonian propagation, DiffnConstraint compulsory-part propagation, and RegularConstraint forward-backward DP propagation in a combined fixpoint loop via `PROPAGATORS` list; each propagator can enable the others to make further reductions. Many highly-constrained problems (Zebra, Sudoku, MagicSquare) are solved entirely at this step. Adding a new propagator requires one line: `FixpointConsistency.of(MyConstraint.class)`. The `snap` field controls `BoundedDomain` handling: `true` (satisfaction chain) snaps non-singleton intervals to their midpoint giving one concrete solution; `false` (optimization chain) leaves intervals open for `BisectionConditioningSolver` downstream.
   - **Dirty-variable seeding (2026-07-16)** — `applyFixpoint(csp, initialSeed)` threads an optional `Set<Variable<?>>` hint through `PROPAGATORS` via `ConstraintConsistency.apply(csp, Set)` (see that interface below); only `NogoodFixpointConsistency` (the `NogoodConstraint` entry, replacing a plain `FixpointConsistency.of(NogoodConstraint.class)`) acts on it, skipping nogoods whose variables are all provably unchanged since the hint was computed. `Solver.Factory.FULL_PROPAGATION_INFERENCE` seeds round 1 with the diff between pre- and post-MAC domains (`PropagationFixpointSolver.changedVariables`, package-private) rather than `null`, since at a search node the input CSP is exactly the parent's already-converged state; `applyFixpoint`'s own no-seed overload (`null`, used for preprocessing before any search node exists) still falls back to a full first-round scan. Benchmarked (`NogoodPropagationBenchmark`) to have no measurable effect on its own — the actual per-node bottleneck turned out to be `NogoodStore`/`ConstraintSatisfactionProblem`'s copy-and-merge cost, not the number of nogoods propagated — but is kept as a correct, low-risk optimization; see the `NogoodStore` bullet above for the fix that did matter.
3. **`BisectionConditioningSolver`** — only in the optimization chain; handles `BoundedDomain` variables that remain non-singleton after propagation by recursively bisecting the widest interval at its midpoint, re-propagating `SumConstraint`/`LinearConstraint` bounds on each half, and snapping to the midpoint once the width falls within `epsilon` (`Solver.Factory.DEFAULT_BISECTION_EPSILON = 1e-3`). When `findWidestBounded` returns null (all bounded domains are singleton), checks `isFullyDetermined()`: if true returns `forcedSolution(csp)`; if false (discrete variables remain) delegates to the inner chain. Passes through entirely for discrete CSPs (no bounded domains). Streams feasible points filtered by improving objective.
4. **`IndependentSubproblemSolver`** — decomposes into independent subproblems and combines solutions (satisfaction chain only)
5. **`TreeDecompositionSolver`** — applies tree decomposition for near-tree problems; skipped when constraint graph minimum degree ≥ targetTreewidth (exact early exit)
6. **`CutsetConditioningSolver`** — handles cyclic graphs by conditioning on a cycle cutset
7. **`TreeSolver`** / **`DomWdegLubySearch`** — terminal solvers for tree-structured or general CSPs respectively; `BranchAndBoundSolver` implements its own self-contained recursive search directly (not a `SolverDecorator`) rather than delegating to `BacktrackingSearch`, since incumbent-based branch-and-bound pruning needs to interleave with the recursion itself. Since 2026-07-18, `BranchAndBoundSolver` also folds a `NogoodStore` into its own search (see below) — the same CDCL mechanism `DomWdegLubySearch` uses, not something unique to the satisfaction chain any more

`SolverDecorator.getSolutions()` short-circuits immediately when any preprocessing step reduces all domains to singletons, returning the forced assignment without invoking downstream stages.

`DomWdegLubySearch` — the terminal solver in the satisfaction chain — combines two complementary techniques: **dom/wdeg variable ordering** (Boussemart et al. 2004) and **Luby restarts**. Each constraint starts with weight 1; when inference-driven propagation causes a domain wipeout the weights of all active constraints on the failing variable are incremented. The selector picks `argmin(domainSize / weightedDegree)`, where weighted degree is the sum of weights of constraints involving the variable that also involve at least one other unassigned variable (variables with no active constraints get `MAX_VALUE` and are chosen last). It also carries a `NogoodStore` (a `@Builder` field defaulting to a fresh one) and an `Inference` (mandatory, no default — see `Solver.Factory.nogoodLearningInference` below for how `Solver.Factory` picks one): both `searchStream` and `searchOne` compute `cspWithNogoods = nogoodStore.apply(csp)` for each candidate value and use it for `next.isConsistent`, then delegate to a private `inferOrExplain` helper — folding every nogood learned anywhere so far into the constraint set actually being propagated against, rather than checking the candidate assignment against a separate map. `inferOrExplain` calls `inference.applyWithReason(cspWithNogoods, variable, next)`; on infeasibility it calls `selector.incrementWeights(cspWithNogoods, variable, next)`, then, if the returned `ConsistencyResult#reason()` is non-null, records it in `nogoodStore` and fires `Statistics#incrementNogoodsLearned`, before incrementing `backtracks` either way — `reason()` is `null` only when the configured `Inference` deliberately opts out of explaining (see `Inference.withoutReasonTracking`). `Solver.Factory` wires a single `NogoodStore` per top-level search, so restarts share learned nogoods. `getSolutions()` returns a complete lazy stream of all solutions with dom/wdeg ordering and weight accumulation; when limits are hit in `searchStream`, `limits.markLimitReached(stats)` is called and `Stream.empty()` is returned (silent truncation). `getSolution()` additionally applies Luby restarts — the failure budget per restart follows the sequence 1, 1, 2, 1, 1, 2, 4, … (multiplied by `DEFAULT_LUBY_UNIT = 100`) and weights (and the shared `NogoodStore`) are preserved across restarts; when `SolverLimits` are hit in `searchOne`, a `Statistics` snapshot is captured via `limits.markLimitReached(stats)` and `LimitsExceeded` (internal pre-allocated sentinel) is thrown, which `getSolution()` catches and re-throws as `LimitExceededException`; `Optional.empty()` is returned only when the problem is genuinely UNSAT. That snapshot is `next.getStatistics()` itself (the real, live-accumulated object for the current restart) with only `nodesExplored` overwritten to the cross-restart cumulative count (tracked separately via `long[] totalNodes`, since `nodesExplored` resets every restart along with the fresh `Assignment.empty()`) — until 2026-07-16 this constructed a brand-new, zeroed `Statistics` instead, silently discarding whatever real `backtracks`/`nogoodsLearned`/`constraintChecks` had already accumulated that restart; `searchStream`'s equivalent limit-hit path (behind `getSolutions()`) never had this bug, since it always passed `next.getStatistics()` straight through — found via `NogoodPropagationBenchmark`'s forced-truncation scenarios reporting `backtracks=0`/`nogoodsLearned=0` despite hundreds of nodes and real nogood-learning clearly having happened. `BacktrackingSearch` is a standalone, independently-tested generic backtracking search (`BacktrackingSearchTest`) that is not currently wired into any production solver chain — `BranchAndBoundSolver` does not wrap it; it implements `Solver` directly with its own recursive search, so it can interleave incumbent-based pruning (`objective(partial) >= incumbent`) with the recursion itself. (Until 2026-07-06, `Solver.Factory`'s optimization chain constructed a `BacktrackingSearch` solely to satisfy `BranchAndBoundSolver`'s inherited `SolverDecorator.inner` field, which `BranchAndBoundSolver` never actually called — that dead wiring was removed along with the unnecessary `SolverDecorator` inheritance.)

`BranchAndBoundSolver` (2026-07-18) mirrors this same `NogoodStore`/`Inference#applyWithReason` wiring in its own `searchValues`/`inferOrExplain` pair — a separate, independent implementation (its own recursive-search shape doesn't fit `DomWdegLubySearch`'s stream-based one, so there's no shared code, just the same pattern), not shared logic: each candidate value is checked and propagated against `nogoodStore.apply(csp)` rather than the bare `csp`, and `inference.applyWithReason`'s reason (when one can be derived) is recorded back into the store on infeasibility, exactly like `DomWdegLubySearch#inferOrExplain` above. This is orthogonal to B&B's own incumbent-bound pruning (`objective(partial) >= incumbent`): a nogood records a genuine constraint violation, permanently true regardless of the incumbent, while the bound cut records a cost dominance relative to the *current* incumbent — the two prunings compose freely. `Solver.Factory`'s optimization-chain wiring constructs a `NogoodStore.forProblem(csp)` for it (one per top-level optimization search, not per sub-problem — the optimization chain has no `IndependentSubproblemSolver` stage) and picks its `Inference` via the same `nogoodLearningInference(config)` helper the satisfaction chain uses (extracted specifically to avoid duplicating that choice across both chains), so `SolverConfig.nogoodLearningEnabled` now controls CDCL for both chains uniformly — it is no longer true that "the optimization chain doesn't do nogood learning at all."

**Conflict explanation** — until 2026-07-18 this lived behind a separate `ConflictExplainer` `@FunctionalInterface` (`MacAndFixpointConflictExplainer.INSTANCE` its sole production implementation) that re-derived a nogood via a second, from-scratch traversal *after* `Inference.apply` had already found the same failure once — re-running MAC and the entire propagator fixpoint, unseeded, on every wipeout. That split was pure duplication (`ConflictExplainer`'s only real implementation mirrored `Inference` so closely there was no genuine pluggability being bought by keeping them separate), confirmed via `NogoodPropagationBenchmark`: isolating pure explanation cost at matched node counts, it added 25-90% overhead across every UNSAT scenario for zero benefit. So `ConflictExplainer`, `NullConflictExplainer`, and `MacAndFixpointConflictExplainer` are deleted; `Inference#applyWithReason` (see Core Abstractions above) is now the sole mechanism, and every propagation layer that can explain a wipeout does so as a byproduct of the *same* pass that detects it, not a replay — `FixpointConsistency`/`AC3`/`NogoodFixpointConsistency` each call their underlying `propagate`/`revise` exactly once per constraint/arc (identical cost to the feasible path) and only compute a reason at the exact point a wipeout is found.

`ConstraintConsistency.explainConflict` (`Optional<NogoodConstraint>`, default `Optional.empty()`) is now a thin wrapper kept for direct callers/tests and for `ConstraintConsistency`'s own default `applyWithReason` fallback; the real per-pass mechanism is `applyWithReason(csp, changedSinceLastRun) → ConsistencyResult`, which `FixpointConsistency`, `AC3`, and `NogoodFixpointConsistency` all override with a genuine single traversal (`NodeConsistency` relies on the defaults, kept covered by a direct unit-test call, since no other implementor falls back to them any more). `FixpointConsistency#applyWithReason` (2026-07-12) tries two tiers per failing constraint: (1) the constraint's own `Propagatable#explainInfeasible` — tightest when it applies (e.g. a specific ground value, or `AllDiffConstraint`'s Hall-set subset); (2) `RangeNogoodConstraint#fromCurrentBounds` over that same constraint's own variables — sound whenever (1) is empty, since `propagate` already reported infeasibility given exactly those current domains (that's the whole of what's needed; no propagator-specific reasoning), just not always as tight as (1) would be if it applied. Earlier (feasible-step) reasons are never accumulated across constraints on the way to a wipeout — only the terminal infeasible step's own two-tier result ever contributes anything. `AC3#applyQueueWithReason` is deliberately ground-only, no tier 2: its wipeout is a per-arc support-existence check on `revise`, not a single constraint's own `propagate` call, so the tier-2 soundness argument (which rests specifically on `Propagatable#propagate`'s infeasibility contract) doesn't transfer without separate justification not attempted here. `NogoodFixpointConsistency#applyWithReason` needs only one tier: every `NogoodConstraint` implementation's `explainInfeasible` unconditionally returns `Optional.of(this)`, so `RangeNogoodConstraint#fromCurrentBounds` is never needed as a fallback there. `Solver.Factory.FULL_PROPAGATION_INFERENCE` is the sole production `Inference` combining these: `MAC#applyWithReason` followed by `PropagationFixpointSolver#applyFixpointWithReason` (itself iterating each `PROPAGATORS` entry's own `applyWithReason`), falling back to the current assignment as a `GroundNogoodConstraint` whenever neither layer derives a reason of its own — so it never leaves a wipeout unexplained. See the `NogoodStore` bullet above for how the resulting nogoods get folded back into the CSP; a single `NogoodStore` instance is shared across Luby restarts (and, for `BranchAndBoundSolver`, across the whole optimization search) so learned nogoods compound.

`NogoodConstraint` (`constraints.nary` package) is an interface (`extends Constraint, Propagatable`) modelling a learned nogood — a specific combination of variable states known to be jointly infeasible — as an actual constraint that joins the same propagation fixpoint as every other constraint, rather than as a separate variable-value map matched against the search's explicit assignment (the design used before, 2026-07-05). This is deliberate: a plain assignment-map comparison can only ever match variables that appear in the search's own explicit decision map, but a variable forced to a singleton value purely by *another* constraint's propagation (never explicitly branched on by search) never appears there, no matter how sound the nogood citing it is — making it permanently unmatchable in any future branch. Modelling the nogood as a `Propagatable` constraint sidesteps this: `propagate()` reasons from current domain state directly (not `Assignment` values), so it catches exactly the "forced, never branched" case a map comparison couldn't. Being an interface (2026-07-12) rather than a single class lets different derivation strategies produce different nogood *shapes* without `NogoodStore`/`ConstraintSatisfactionProblem` (both of which only depend on the interface) needing to change. Registered once, for the whole interface, in `PropagationFixpointSolver.PROPAGATORS` via `NogoodFixpointConsistency.INSTANCE` (2026-07-16; previously a plain `FixpointConsistency.of(NogoodConstraint.class)`, replaced because the nogood set — unlike every other propagator's fixed, small constraint count — grows unboundedly over a search, up to `NogoodStore`'s `20 * variableCount` cap, making a full rescan of every nogood on every fixpoint round the dominant cost in long searches) — `DomWdegVariableSelector` (`instanceof NogoodConstraint`, excluding it from dom/wdeg weighting) dispatches via `isInstance`, which works polymorphically against interfaces, so any implementation is picked up automatically. Two implementations exist:

- **`GroundNogoodConstraint`** — forbids one specific ground value per variable: the clause `OR(x1 != v1, ..., xk != vk)` over its `forbidden: Map<Variable<?>, Object>` map. `propagate()` classifies each `(variable, forbidden value)` pair against current domain state via the generic `Domain#contains`/`Domain#isSingleton` — *satisfied* (domain no longer contains the value — the literal is permanently true, so the whole clause is satisfied), *falsified* (domain is exactly `{value}`), or *undetermined* (domain still contains it alongside others) — and infers accordingly: all falsified → infeasible; exactly one undetermined with the rest falsified → unit-propagate by pruning that value (only for `DiscreteDomain`; a single point removed from a non-singleton bounded/continuous domain isn't a meaningful narrowing, so that case is left untouched). `explainInfeasible` simply returns `Optional.of(this)` — a falsified ground clause is always its own sound explanation, no singleton gating needed (that requirement lives one level up, in whichever propagator derived `forbidden` in the first place). A static `fromReason(Map<Variable<?>, Object> reason)` helper (2026-07-12, since `Propagatable.explainInfeasible` returns `Optional<NogoodConstraint>` — see below) wraps a raw reason map for every propagator's own `explainInfeasible`: an empty map becomes `Optional.empty()` (no sound reason), a non-empty one becomes `Optional.of(GroundNogoodConstraint.of(reason))` — every propagator-specific `explainInfeasible` in this file funnels its raw `Map<Variable<?>, Object>` reason through this one call rather than constructing `GroundNogoodConstraint` directly. `of()` asserts the `forbidden` map is non-empty — an empty map is the empty clause, which `propagate()` reports infeasible for every state (`undeterminedCount == 0`) and would prune the entire search; `fromReason` guards against ever calling `of()` with one.
- **`RangeNogoodConstraint`** (2026-07-12) — generalises the above to forbid a whole numeric range per variable: `OR(x1 ∉ R1, ..., xk ∉ Rk)` over its `forbidden: Map<Variable<?>, IntervalDomain>` map, reusing `IntervalDomain` itself as the forbidden-region representation. A literal is *satisfied* once the domain no longer overlaps its forbidden range at all, *falsified* once the domain is entirely contained within it (no singleton requirement, unlike `GroundNogoodConstraint`), or *undetermined* on partial overlap; classification is exact enumeration for `DiscreteDomain` and bounds comparison (via `NumericBounds`) for `BoundedDomain`. Unit propagation on the sole undetermined literal prunes the *whole* forbidden range in one step — for a `DiscreteDomain`, every forbidden value is deleted; for a `BoundedDomain`, only an overlap touching one edge of the domain can be narrowed (to a single interval), since `BoundedDomain` can't represent a "hole" left by an interior overlap — that case is left untouched, same principle as `GroundNogoodConstraint`'s non-singleton-bounded skip. `explainInfeasible` simply returns `Optional.of(this)`, same reasoning as `GroundNogoodConstraint`: a falsified range clause is unconditionally its own sound explanation — no singleton requirement, unlike citing individual ground values would need, since "this variable's whole current domain is forbidden" is exactly what falsified already established. Also has a static `fromCurrentBounds(variables, domains)` factory: cites every one of `variables`' *current* domain bounds (a degenerate point range for an already-singleton one), intended to be sound whenever the constraint they jointly belong to has just reported infeasible via `Propagatable#propagate` — that return value alone already means no combination drawn from those exact domains satisfies it, no propagator-specific reasoning needed *in the ground case*. But citing an interval `[min,max]` as a stand-in for a **gapped** discrete domain is unsound: a later, unrelated domain nested inside that interval (but never actually part of the original domain) would satisfy "contained in `[min,max]`" without ever being a subset of the domain that caused the original infeasibility, silently generalising the nogood past what was proven. This is exactly what broke when `AllDiffConstraint` started citing Hall-violating subsets (which are gapped almost by definition) through this method directly instead of via `FixpointConsistency`'s whole-constraint-variable-set fallback: `NQueensTest` silently dropped from 92 solutions to 80 (2026-07-13) before a private `isSafeToCiteAsRange` gate was added — true for any `BoundedDomain` (continuous, so min/max already is the whole domain, no gap possible) but for a `DiscreteDomain` only when every value is an `Integer` *and* the domain is gapless (`size() == max - min + 1`, i.e. the interval and the domain denote exactly the same set of values); otherwise `fromCurrentBounds` returns `Optional.empty()`. This is `FixpointConsistency#explainConflict`'s tier-2 fallback (see the "Conflict explanation" section above) and `AllDiffConstraint`'s own second tier (see below) — the two mechanisms that actually wire `RangeNogoodConstraint` into real search, not the `RangeNogoodConstraint.of(Map<Variable<?>, IntervalDomain>)` constructor itself, which no derivation strategy calls directly.

Both implementations are registered in `ConstraintSatisfactionProblem.CONTINUOUS_COMPATIBLE_CONSTRAINTS` by concrete class, not the `NogoodConstraint` interface — that check matches on `getClass()`, so any future additional implementation needs its own entry there too (a nogood can legitimately cite a variable that's a `BoundedDomain`, e.g. one already snapped by the satisfaction chain, before search ever begins).

There is deliberately no separate "nogood prunes" statistic: since a nogood-caused rejection is now indistinguishable from any other constraint-caused rejection at `isConsistent`/`inference.apply` (that's the point — nogoods are first-class constraints, not a bolted-on special case), it's counted under `backtracks` like everything else; distinguishing it would require an expensive double-check with no correctness benefit.

`ArcConsistentSolver`, `AllDiffConsistentSolver`, and `CumulativeConsistentSolver` have been deleted — their functionality is covered by `AC3.INSTANCE` and `FixpointConsistency.of(...)` entries in `PROPAGATORS`.

### Local Search Chain

`LocalSolver.Factory.INSTANCE.createLocalSolver(maxAttempts, maxSteps, factory)` builds:

```
NodeConsistency → UnaryComparatorBounds → BinaryComparatorBounds → OffsetBounds → AC3 → SumBounds → LinearBounds → CountValue → InverseArc → AmongValue → AtLeastN/AtMostN → CumulativeTimetable → GlobalCardinalityValue → LexBounds → TuplesGAC → IndependentSubproblems → MinConflicts
```

Seeded by `RandomAssignmentFactory`, `GreedyAssignmentFactory`, or `FallbackAssignmentFactory`.

`MinConflictsSolver`, `TabuSearchSolver`, and `WalkSATSolver` each run all `maxAttempts` restarts in parallel via `IntStream.parallel()`: the satisfaction path returns the first solution found (unordered); the optimization path runs all attempts in parallel and returns the true global minimum (MinConflicts and TabuSearch only — WalkSAT does not support optimization). `IndependentSubproblemLocalSolver` uses `parallelStream()` so disjoint subproblems are also solved concurrently.

`LocalSolver.Factory.INSTANCE` automatically routes to `WalkSATSolver` for the satisfaction overload when all variable domains are boolean (`BooleanDomain` or `DiscreteDomain` containing only booleans) AND the CSP contains no `ExactlyOneConstraint` or `AtLeastNConstraint` (which require two-step moves that WalkSAT handles poorly). Falls back otherwise to `raced` — `RaceLocalSolver` running `MinConflictsSolver` against `TabuSearchSolver` (see below), wrapped in `IndependentSubproblemLocalSolver`. The objective overload routes to `LargeNeighborhoodSolver` when the reduced CSP contains any `ExactlyOneConstraint` (LNS's destroy-repair moves over exactly-one slots outperform MinConflicts for these problems); falls back to that same `raced` otherwise.

`RaceLocalSolver` races an arbitrary list of `LocalSolver` delegates (currently `MinConflictsSolver` + `TabuSearchSolver`) against the same problem on virtual threads and returns whichever produces a result first — sidestepping the need for a heuristic to predict which strategy suits a given problem shape (a router built on such a heuristic was tried and falsified before for a different pair of solvers; see `BacktrackingSearch` vs `DomWdegLubySearch` in project history). Delegates implementing the package-private `CancellableLocalSolver` contract (`MinConflictsSolver`, `TabuSearchSolver`) are handed a shared `Cancellation` token, tripped as soon as any delegate wins, so losers stop at their next search step rather than running to completion for no benefit; other delegate types just finish in the background once they've lost. A delegate returning `Optional.empty()` doesn't end the race — only once every delegate has finished without a result. (2026-07-18) `race`'s exception handling rethrows a delegate's `RuntimeException` or `Error` as its real type rather than unconditionally casting to `RuntimeException`, which previously turned an `Error` (e.g. an assert failure elsewhere in the solver chain) into a misleading `ClassCastException` masking the real failure.

`TabuSearchSolver` — the same min-conflicts move selection as `MinConflictsSolver`, plus a short-term memory (`tabuTenure`, default 10 steps) that forbids reverting a variable to the value it just held, unless doing so would strictly improve on the best total conflict weight seen so far this attempt (the aspiration criterion — see `feedback_aspiration_needs_global_cost` in project memory: it must compare total assignment cost, not local per-move cost). Total weight is maintained incrementally off each move's own delta rather than rescanned from the whole CSP on every candidate, keeping per-step cost the same order as `MinConflictsSolver`'s.

`LargeNeighborhoodSolver` — destroy-repair local search for boolean CSPs with `ExactlyOneConstraint`s. Each step picks `slotsPerStep` (default 2) random exactly-one slots, enumerates all valid refill combinations (exactly one variable true per slot), and accepts the combination with the fewest violations (ties broken by objective). Runs all `maxAttempts` restarts in parallel; supports both satisfaction and optimization. Wrapped in `IndependentSubproblemLocalSolver` like other terminal solvers. `pickSlots` shuffles via `ThreadLocalRandom.current()` directly rather than constructing a fresh reseeded `Random` per call.

The preprocessing chain before `MinConflicts` runs all `Propagatable` constraint types (via `PREPROCESSORS` list in `LocalSolver.Factory`) in the same order as `PropagationFixpointSolver.PROPAGATORS` except without the outer fixpoint loop and without AllDiff GAC. Includes `AtLeastNConstraint` and `AtMostNConstraint` propagation — particularly useful for local search where these boolean constraints have no binary decomposition to rely on. This detects infeasibility early (avoiding wasted attempts) and prunes domains to improve initial assignment quality. AllDiff GAC is intentionally excluded — it is expensive and repair-based search does not benefit from that level of arc consistency.

**Conflict-scoring completeness (2026-07-18)** — `LocalSearchSupport.conflictConstraints` (package-private, shared by `MinConflictsSolver`/`TabuSearchSolver`/`WalkSATSolver`) drives conflict-repair move scoring (`weighConflicts`/`totalWeight`/`incrementViolatedWeights`): it prefers a constraint's `BinaryDecomposable` decomposition over the constraint itself wherever available, for per-pair scoring granularity. This was previously unconditional, which was unsound for `ExactlyOneConstraint` (whose inherited pairwise-NAND decomposition only rules out two-true, never zero-true) and `CircuitConstraint` (whose all-different decomposition only rules out repeats, never sub-tours): a search could reach, e.g., every variable in an ExactlyOne slot false and see zero tracked violations, since neither gap's decomposition alone can express the missing whole-constraint condition. `Assignment#isSolution` (the actual accept/reject check) always used the real constraint directly, so this never caused a wrong final answer — but `weighConflicts` had no signal to move away from such a state, degrading `MinConflictsSolver`/`TabuSearchSolver` to random tie-breaking on that dimension instead of directed repair. Fixed by adding `BinaryDecomposable#isDecompositionComplete()` (see below) and having `conflictConstraints` keep the original n-ary constraint alongside an incomplete decomposition rather than dropping it. This also made a previously-reachable `WalkSATSolver` branch (`if (unsatisfied.isEmpty()) break;`) provably dead code — `conflictConstraints` is now a faithful stand-in for the CSP's true satisfiability, so that check could only ever have fired at a point where `isSolution(csp)` had already been checked false for the same assignment moments earlier, a contradiction; the branch was only ever reachable via this exact gap (confirmed by the one existing test that hit it, which calls `WalkSATSolver` directly — bypassing `Factory` routing — on an `ExactlyOneConstraint` CSP) and was removed rather than left permanently uncovered.

### Constraint Construction

`CSP.Builder` provides fluent helper methods. All binary constraint classes also have static `of()` factory methods.

**Unary**
```java
csp.equalsConstraint(v, value)
csp.notEqualsConstraint(v, value)
csp.predicateConstraint(v, predicate)
csp.comparatorConstraint(v, Operator.GEQ, value)   // v >= value (Number types)
```

**Binary**
```java
csp.equalsConstraint(v1, v2)
csp.notEqualsConstraint(v1, v2)
csp.notEqualsChainConstraint(List.of(v1, v2, v3))  // consecutive pairs differ
csp.offsetConstraint(v1, offset, Operator.EQ, v2)  // v1 + offset == v2
csp.comparatorConstraint(v1, Operator.LEQ, v2)     // v1 <= v2 (any Comparable)
csp.logicConstraint(b1, LogicOperator.OR, b2)       // boolean connective (AND/OR/XOR/NAND/NOR/XNOR)
csp.elementConstraint(index, result, array)          // result = array[index] (1-based); array is a fixed List<T>
csp.elementVariableConstraint(index, result, vars)   // result = vars[index] (1-based); vars is a List<Variable<T>>
csp.biPredicateConstraint(v1, v2, predicate)
```

**N-ary**
```java
csp.allDiffConstraint(Set.of(v1, v2, v3))
csp.atMostOneConstraint(Set.of(b1, b2, b3))        // AC3 decomposition into BinaryLogicConstraint(NAND) plus its own counting propagation
csp.atMostNConstraint(Set.of(b1, b2, b3), n)
csp.atLeastNConstraint(Set.of(b1, b2, b3), n)      // prefer for local search
csp.atLeastNConstraintWithCounting(Set.of(b1, b2, b3), n)  // prefer for backtracking (carry-chain)
csp.exactlyOneConstraint(Set.of(b1, b2, b3))       // also forces the sole remaining candidate true once every other is excluded
csp.sumConstraint(Set.of(v1, v2, v3), Operator.EQ, 10)
csp.maxConstraint(Set.of(v1, v2, v3), Operator.LEQ, 10)
csp.minConstraint(Set.of(v1, v2, v3), Operator.GEQ, 0)
csp.productConstraint(Set.of(v1, v2, v3), Operator.EQ, 24)  // v1*v2*v3==24; propagates for EQ/LEQ/GEQ when all domains have strictly positive mins
csp.divisionConstraint(dividend, divisor, Operator.EQ, 3)   // dividend/divisor==3; propagates for EQ/LEQ/GEQ when both domains have strictly positive mins
csp.linearConstraint(Map.of(v1, 2, v2, 3), Operator.LEQ, 10)  // weighted sum
csp.countConstraint(Set.of(v1, v2, v3), value, Operator.EQ, 2)
csp.amongConstraint(Set.of(v1, v2, v3), Set.of(a, b), Operator.EQ, 2)  // count vars with value in {a,b}
csp.inverseConstraint(List.of(f1, f2, f3), List.of(g1, g2, g3))        // f[i]==j ↔ g[j-1]==i+1
csp.globalCardinalityConstraint(Set.of(v1, v2, v3), Map.of(a, 2, b, 1))
csp.cumulativeConstraint(starts, durations, resources, limit)  // resource scheduling
csp.tuplesConstraint(Set.of(Assignment.of(...), ...))          // extensional (table)
csp.increasingConstraint(List.of(v1, v2, v3))      // v1 <= v2 <= v3; AC3 decomposition
csp.decreasingConstraint(List.of(v1, v2, v3))      // v1 >= v2 >= v3; AC3 decomposition
csp.lexConstraint(List.of(a1, a2), Operator.LEQ, List.of(b1, b2))
csp.predicateConstraint(Set.of(v1, v2, v3), predicate)
csp.circuitConstraint(List.of(s0, s1, s2))                         // Hamiltonian circuit; successors.get(i) = 1-indexed successor of node i+1
csp.diffnConstraint(xs, ys, widths, heights)                        // pairwise non-overlapping 2D rectangles; origin variables may be IntRangeDomain or IntervalDomain
csp.regularConstraint(sequence, automaton)                          // DFA-constrained sequence; values must spell a word accepted by the Automaton
```

**Reification**
```java
csp.reifyConstraint(b, constraint)    // b <-> constraint
csp.impliesConstraint(b, constraint)  // b -> constraint
```

### Key Conventions

- **Immutability**: `Assignment`, `Variable`, and constraint objects use Lombok `@Value`; `CSP` uses `@Builder`/`@Singular`. Constraint subclasses use `@SuperBuilder` + `@EqualsAndHashCode(callSuper = true)`. All domain classes are records: `IntervalDomain(double min, double max)`, `DomainObjectSet<T>(Set<T> values)`, `IntRangeDomain(Set<Integer> values)`, `EnumDomain<E>(Set<E> values)`, `BooleanDomain()`, `AssignedDomain(Object value)`, `AssignmentDomain(Set<Assignment> values)`.
- **Lombok**: `@Value`, `@Builder`, `@SuperBuilder`, `@Singular`, `@Slf4j` are used extensively — do not add manual boilerplate that Lombok already provides
- **Static factories**: All constraint classes have a static `of()` factory method; use these instead of `.builder()...build()` in production code
- **Null safety**: JSpecify `@NonNull`/`@Nullable` annotations throughout; `Optional` used for nullable returns
- **Logging**: All solvers/consistency algorithms use `@Slf4j` (SLF4J) for debug/info logging
- **Assertions**: Preconditions (e.g., equal list sizes) are checked with Java `assert` statements
- **`Operator` enum** — in `constraints` package; covers EQ, NEQ, LT, GT, LEQ, GEQ
- **`LogicOperator` enum** — in `constraints` package; covers AND, OR, XOR, NAND, NOR, XNOR
- **`ConstraintConsistency` interface** — `@FunctionalInterface` in `consistency` package; the common contract for all propagation/consistency passes: `apply(ConstraintSatisfactionProblem) → Optional<ConstraintSatisfactionProblem>`. Implemented by `AC3`, `NodeConsistency`, and any `FixpointConsistency.of(...)` instance. `PropagationFixpointSolver` iterates a static `PROPAGATORS` list; `LocalSolver.Factory` iterates a static `PREPROCESSORS` list — adding a new propagator to either chain is a one-line `FixpointConsistency.of(MyConstraint.class)` entry. Also declares a default `explainConflict(ConstraintSatisfactionProblem) → Optional<NogoodConstraint>` returning `Optional.empty()`, now a thin wrapper kept for direct callers/tests (see the "Conflict explanation" section above); the real per-pass explanation mechanism is `applyWithReason(csp, changedSinceLastRun) → ConsistencyResult` (2026-07-18, replacing the deleted `ConflictExplainer`), which `FixpointConsistency`, `AC3`, and `NogoodFixpointConsistency` all override with a genuine single traversal; `NodeConsistency` relies on both defaults (kept covered by a direct unit-test call, since no other implementor falls back to them any more). A second default, `apply(ConstraintSatisfactionProblem, Set<Variable<?>> changedSinceLastRun) → Optional<ConstraintSatisfactionProblem>` (2026-07-16), delegates to the single-arg `apply` by default; only `NogoodFixpointConsistency` overrides it meaningfully (see below and the `PropagationFixpointSolver` bullet above) — every other implementor's constraint count is fixed and small, so the hint would have nothing worth skipping.

  `AC3#applyQueueWithReason`'s Javadoc covers the ground-only soundness argument for why citing only `X_j` would be unsound (an earlier attempt got this wrong and silently over-pruned NQueens from 92 solutions to 4). The frozen-domains hang this class had until 2026-07-16 (`revise` reading a stale `problem` snapshot instead of the caller's progressively-narrowed map, letting dense/cyclic binary CSPs loop forever) is likewise documented on the private `revise(Map, Arc, BinaryConstraint)` overload.
- **`FixpointConsistency`** — concrete final class in `consistency.fixpoint` package; implements `ConstraintConsistency` for the common pattern of filtering constraints by type and running them to fixpoint. Created via `FixpointConsistency.of(XxxConstraint.class)`; logs under `FixpointConsistency` using the constraint type's simple name in the message. Replaces the deleted per-type consistency classes and the deleted `ConsistencyFixpoint` utility class. Overrides `explainConflict` with the two-tier logic described in the "Conflict explanation" section above (own `Propagatable#explainInfeasible` first, `RangeNogoodConstraint#fromCurrentBounds` as fallback); returns `Optional.empty()` immediately if the CSP has no constraints of its `constraintType`.
- **`NogoodFixpointConsistency`** — concrete final class in `consistency.fixpoint` package (2026-07-16), sibling to `FixpointConsistency` but specialized for `NogoodConstraint`: reads `csp.getNogoods()` directly (not `getConstraints()`, avoiding a type-filter scan of the flat structural+nogood union). Overrides `apply(csp, changedSinceLastRun)` to skip any nogood whose variables are all absent from the hint, sound because a `NogoodConstraint`'s `propagate()` result depends only on its own variables' current domains — a `null` hint (unknown, e.g. round 1 of a fixpoint call with no parent to diff against) falls back to a full scan, same as `FixpointConsistency`. `explainConflict` always scans every nogood (cold path, only reached once a conflict is already known) and needs only one tier, not `FixpointConsistency`'s two: every `NogoodConstraint` implementation's `explainInfeasible` unconditionally returns `Optional.of(this)`, so `RangeNogoodConstraint#fromCurrentBounds` is never needed as a fallback here.
- **`Propagatable` interface** — in `consistency` package; constraints that support domain propagation implement `propagate(Map<Variable<?>, Domain<?>> domains) → Optional<Map<Variable<?>, Domain<?>>>`. Also declares a default `propagateWithReasons(Map<Variable<?>, Domain<?>> domains) → PropagationResult` that delegates to `propagate`; on infeasibility it delegates further to a default `explainInfeasible(Map<Variable<?>, Domain<?>> domains) → Optional<NogoodConstraint>` method, which returns `Optional.empty()` (no explanation; callers substitute the full assignment as the nogood) unless a constraint overrides it with a tighter, sound explanation. This feasible-path wrapping (`propagate().map(...).orElseGet(...)`) is identical for every implementer and lives once in `propagateWithReasons`'s default body, so overriders only ever implement `explainInfeasible`. Most overriders build a raw `Map<Variable<?>, Object>` reason internally and funnel it through `GroundNogoodConstraint.fromReason` at the very end — `AllDiffConstraint` is the one exception, since it also has a `RangeNogoodConstraint` tier to fall back to. Two `static` helpers on `Propagatable` factor out the two recurring explanation shapes, both called via the fully-qualified `Propagatable.xxx(...)` form since static interface methods aren't inherited by implementors in Java:
  - **`addIfSingleton(domain, variable, reason)`** — "blame the singleton side" for binary constraints: if `domain` is a singleton, records its sole value against `variable`; a non-singleton side is left unblamed since no single value can be cited for it. Used by `BinaryComparatorConstraint`, `BinaryOffsetConstraint`, `AbsoluteDifferenceConstraint`, and `DivisionConstraint` — see each class's own `explainInfeasible` Javadoc for why a two-sided bounds violation only ever needs the singleton side(s).
  - **`allSingletonReason(variables, domains)`** — "fully collective" explanation: returns a full variable-value reason map only when *every* one of `variables` currently holds a singleton domain, or `Map.of()` otherwise, since a partial subset can't rule out an unlisted open-domain variable also participating in the violation. Used by `SumConstraint`, `LinearConstraint`, `ProductConstraint`, `NaryTuplesConstraint`, `InverseConstraint`, `NaryElementConstraint`, `AllDiffConstraint`, and `RangeNogoodConstraint` — each explains in its own `explainInfeasible` Javadoc why its particular violation shape can't be attributed to any smaller subset.

Detailed per-constraint `propagate()`/`explainInfeasible()` algorithm walkthroughs (which bounds pass applies for which operator, which subset of variables is cited and why a smaller one wouldn't be sound, which helpers are shared between the two methods) live in each constraint class's own Javadoc, not here — every `Propagatable` implementation documents its own propagation algorithm and its own explanation soundness argument at the point the code is. A few soundness fixes were caught by real regressions and are worth knowing about even though the reasoning itself is now in code: citing only the singleton subset of `DiffnConstraint`'s four responsible variables (instead of the fully collective `allSingletonReason` set) regressed `RectanglePackingTest` from 12 solutions to 8; `RangeNogoodConstraint#fromCurrentBounds` citing a gapped discrete domain's bounding interval (instead of gating on gaplessness) regressed `NQueensTest` from 92 solutions to 80 once `AllDiffConstraint` started using it for Hall-violating subsets.

Every `Propagatable` constraint overrides `explainInfeasible` except `UnaryComparatorConstraint`, which deliberately never will — see its own Javadoc for why (in short: its infeasibility path is unreachable once search begins, since the satisfaction chain fully resolves or rejects every `BoundedDomain` during preprocessing). `BinaryComparatorConstraint`, `BinaryOffsetConstraint`, `AbsoluteDifferenceConstraint`, `DivisionConstraint`, `AllDiffConstraint`, `SumConstraint`, `LinearConstraint`, `CountConstraint`, `InverseConstraint`, `AmongConstraint`, `AtLeastNConstraint`, `AtMostNConstraint`, `AtMostOneConstraint`, `ExactlyOneConstraint`, `CumulativeConstraint`, `GlobalCardinalityConstraint`, `LexConstraint`, `MaxConstraint`, `MinConstraint`, `NaryElementConstraint`, `NaryTuplesConstraint`, `ProductConstraint`, `CircuitConstraint`, `DiffnConstraint`, `RegularConstraint`, `GroundNogoodConstraint`, `RangeNogoodConstraint`, `IncreasingConstraint`, and `DecreasingConstraint` implement it; each is registered in the solver chains via `FixpointConsistency.of(XxxConstraint.class)`. `AtMostOneConstraint`/`ExactlyOneConstraint` (2026-07-17) and `IncreasingConstraint`/`DecreasingConstraint` (2026-07-18) are the two pairs that implement `Propagatable` *in addition to* an existing `BinaryDecomposable` pairwise decomposition — each class's own Javadoc explains why the decomposition alone was insufficient (a whole-constraint counting condition for the first pair, `BoundedDomain`/fixpoint reach for the second). The `AtMostOneConstraint`/`ExactlyOneConstraint` gap was found via a boolean-encoded pigeonhole scenario added to `NogoodPropagationBenchmark`: before that fix, neither constraint's violations were ever caught by propagation (only by `Assignment#isConsistent`'s direct `isSatisfiedBy` check), so `DomWdegLubySearch` never learned weights or nogoods from pigeonhole failures at all. Two attempts to fix this from the `DomWdegLubySearch` side instead (unconditional weight+nogood learning on `isConsistent` failures; weight-only) were tried first and reverted after regressing `CryptarithmeticTest` ~30x and ~4x in wall-clock respectively, with the weight-only version also making pigeonhole itself worse — see `project_jcsp_isconsistent_learning_gap` in project memory for that trail. Giving the constraints real propagation instead needed no `DomWdegLubySearch` changes and cut pigeonhole's full UNSAT proof from ~8.3s to 2.45-3.19s.
- **`CircuitConstraint`** — enforces a single Hamiltonian circuit through all n nodes; `successors.get(i)` is the 1-indexed successor of node i+1. Implements both `Propagatable` and `BinaryDecomposable` (delegates to `AllDiffConstraint` for AC3 arc pairs since circuit implies all-different, though that decomposition alone misses sub-tours — see `isDecompositionComplete()` below). Factory: `CircuitConstraint.of(List<Variable<Integer>> successors)`. Propagation algorithm (self-loop removal, singleton propagation, sub-tour elimination) and explanation reasoning are on the class's own Javadoc.
- **`DiffnConstraint`** — enforces pairwise non-overlap of axis-aligned rectangles via pairwise compulsory-part reasoning; origin variables may be `IntRangeDomain` or `IntervalDomain`. Added to `CONTINUOUS_COMPATIBLE_CONSTRAINTS`. Factory: `DiffnConstraint.of(xs, ys, widths, heights)`. See its own Javadoc for the propagation algorithm and explanation reasoning.
- **`IncreasingConstraint`/`DecreasingConstraint`** (2026-07-18) — implement `Propagatable` directly, in addition to their pairwise `BinaryDecomposable` decomposition; see the class Javadoc for the bounds-consistency algorithm (`OrderingPropagation` helper) and why it isn't restricted to `Number` the way `BinaryComparatorConstraint`'s decomposition is.
- **`Automaton<T>`** — record (`numStates`, `initialState`, `acceptingStates`, `transitions`) used by `RegularConstraint`; states are integers `0..numStates-1`; `transition(state, value)` returns -1 for missing (dead) transitions; compact constructor asserts all state indices are valid. Factory: `Automaton.of(numStates, initialState, acceptingStates, transitions)`.
- **`RegularConstraint`** — enforces that the sequence of values assigned to `sequence` is accepted by a `DFA`; propagation is full GAC via forward-backward DP. Does not have `@Value` (inherits `NaryConstraint.toString()`). Factory: `RegularConstraint.of(List<Variable<T>> sequence, Automaton<T> automaton)`. See its own Javadoc for the two-pass algorithm and its two distinct infeasibility points.
- **`BinaryDecomposable` interface** — in `constraints` package; n-ary constraints that can be decomposed into an equivalent set of binary constraints implement `getAsBinaryConstraints() → Set<BinaryConstraint<?,?>>`. `AllDiffConstraint`, `AtMostOneConstraint` (and `ExactlyOneConstraint`), `IncreasingConstraint`, `DecreasingConstraint`, `ReifiedConstraint`, and `CircuitConstraint` implement it. `CircuitConstraint` delegates to `AllDiffConstraint.getAsBinaryConstraints()` since circuit implies all-different. Used by `ConstraintGraph` to infer additional binary constraints for AC3, and by `ConstraintSatisfactionProblem`/`MinConflictsSolver` to identify non-decomposable n-ary constraints. `ReifiedConstraint` returns an empty set when its body is not a `UnaryConstraint`.

  - **`isDecompositionComplete()`** (2026-07-18, default `true`) — whether `getAsBinaryConstraints()` alone is a sound *and complete* stand-in for the constraint's own `isSatisfiedBy` (every decomposed binary constraint satisfied iff the original is), not merely a necessary condition. True for `AllDiffConstraint`, `IncreasingConstraint`/`DecreasingConstraint`, and `ReifiedConstraint`. `ExactlyOneConstraint` and `CircuitConstraint` override it `false` — see each class's own Javadoc for what its pairwise decomposition misses. `AC3`/`ConstraintGraph` never consult this flag, since they only ever use a decomposition as *additional* propagation alongside the original constraint, never as its replacement; `LocalSearchSupport#conflictConstraints` (see "Conflict-scoring completeness" above) is the one caller that does, since it substitutes the decomposition *for* the original when scoring conflicts.

### Benchmarking

`io.github.rcrida.jcsp.benchmark.NogoodPropagationBenchmark` (`src/test/java`, plain `main()` class — not a JUnit test, so it isn't run by surefire and doesn't count against the 100%-coverage gate) measures `NogoodStore` overhead across three variants (`default`/`capped`/`disabled`, see the `NogoodStore`/`SolverConfig` bullets above) run against the exact same hard CSP under the same node budget, so any wall-clock gap is attributable to nogood-store/CDCL cost rather than different search decisions. Covers four deliberately different shapes: `golombRuler` (one large `AllDiffConstraint`, broad GAC-driven propagation, UNSAT), `randomBinaryCsp` (a seeded Model-B random binary CSP built entirely from `biPredicateConstraint`, so only `AC3` and nogoods ever propagate, UNSAT), `quasigroupCompletion` (a random Latin-square completion instance near the Gomes/Selman phase-transition region, SAT), and `pigeonhole` (the classic boolean-encoded pigeonhole principle — `exactlyOneConstraint`/`atMostOneConstraint`, deliberately not `allDiffConstraint`, which would let Regin's GAC catch the Hall violation with zero search — UNSAT, resolution-hard per Haken 1985). Run via `mvn test-compile` then `java -cp target/classes:target/test-classes:$(mvn -q dependency:build-classpath -Dmdep.outputFile=/dev/stdout) io.github.rcrida.jcsp.benchmark.NogoodPropagationBenchmark`.

`pigeonhole`'s three variants come out statistically identical (nogood learning has zero effect, not just a small one) — a real, confirmed architectural gap, not a benchmark artifact: `DomWdegLubySearch` only calls `selector.incrementWeights`/records a nogood inside the branch where `inference.applyWithReason` (propagation) reports infeasible, but neither `ExactlyOneConstraint` nor `AtMostOneConstraint` is registered in `PropagationFixpointSolver.PROPAGATORS` (only their pairwise-NAND `BinaryDecomposable` decomposition feeds `AC3`, which can force individual variables but can never detect a whole-constraint violation like "zero holes assigned true" as a domain wipeout — an inherently global/counting condition). Every pigeonhole failure is instead caught by `Assignment#isConsistent`'s direct `isSatisfiedBy` check, which silently bypasses both learning mechanisms regardless of configuration. Two attempted fixes (unconditional weight+nogood learning on `isConsistent` failures too; weight-only) were both empirically reverted (2026-07-17) after regressing `CryptarithmeticTest` ~30x and ~4x respectively in wall-clock, with the weight-only version also making pigeonhole itself worse (stopped finishing its own proof within budget) — see `project_jcsp_isconsistent_learning_gap` in project memory for the full trail. Left as a known, documented gap.

Each run prints `nodesExplored`/`backtracks`/`nogoodsLearned` alongside timing wherever available — but availability depends on how the solve terminates via `DomWdegLubySearch#getSolution` (the method the benchmark calls): SAT and limit-exceeded completions expose real `Statistics`, genuine UNSAT does not (`Optional.empty()` carries no `Statistics` anywhere reachable from that API — a real gap, not a benchmark oversight; scenarios that finish by proving UNSAT under budget report `nodesExplored=n/a`). This surfaced a real bug in `searchOne`'s limit-exceeded path (fixed 2026-07-16): it used to construct a fresh `Statistics` with only `nodesExplored` set, discarding whatever real `backtracks`/`nogoodsLearned`/`constraintChecks` the current restart's `next.getStatistics()` already held at that point — `searchStream`'s equivalent path (behind `getSolutions()`) never had this bug, since it passes `next.getStatistics()` straight through. Fixed by reusing `next.getStatistics()` in `searchOne` too, overwriting only `nodesExplored` with the cross-restart cumulative count. Confirmed via `quasigroupCompletion`: solves in exactly 405 nodes with only 2 backtracks (real search, not zero-node propagation-only solving, but still almost entirely propagation-dominated — the 4+s wall-clock at ~92 nodes/sec reflects 40 `allDiffConstraint`s of size 20 being propagated per node, not search depth) and identical timing across all three variants (nogood learning has no opportunity to help or hurt when there's essentially no backtracking).
